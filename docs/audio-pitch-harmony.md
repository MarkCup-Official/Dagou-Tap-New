# “大狗 Tap”音高变调与和声设计依据

> 更新日期：2026-07-20  
> 用途：统一记录 `da / gou / jiao` 音频变调、音高检测和配合背景和弦时采用的乐理与技术依据。

## 1. 当前音乐结构

项目背景音乐速度为 128 BPM，四个小节循环一次，和弦走向为：

| 小节 | 和弦 | C 大调功能 | 三和弦组成音 |
| --- | --- | --- | --- |
| 1 | C | I | C、E、G |
| 2 | G | V | G、B、D |
| 3 | Am | vi | A、C、E |
| 4 | F | IV | F、A、C |

这是 C 大调中的 `I–V–vi–IV` 进行。三和弦由根音、三音和五音组成；让交互音效落到当前和弦的这三个音上，是获得稳定协和感最直接的办法。

## 2. 五声音阶的适用范围

C 大调五声音阶由以下五个音级组成：

```text
C  D  E  G  A
1  2  3  5  6
```

大调五声音阶使用大调音阶的第 1、2、3、5、6 级，移除了第 4 级和第 7 级。对 C 大调来说，移除的是 F 和 B，因此集合内部没有半音相邻关系，旋律自由触发时通常比完整七声音阶更不容易产生尖锐摩擦。

但“属于 C 大调五声音阶”不等于“在 C–G–Am–F 的每个和弦上都是和弦音”：

- C 和弦的 C、E、G 全部在五声音阶内。
- G 和弦的 G、B、D 中，B 不在五声音阶内。
- Am 和弦的 A、C、E 全部在五声音阶内。
- F 和弦的 F、A、C 中，F 不在五声音阶内。
- 固定使用 C、D、E、G、A 时，部分音在某些和弦上会成为 2、4、6、7 等延伸音或张力音，不能保证每次点击都具有同等稳定度。

因此，本项目若以“尽量怎么按都和谐”为优先目标，应使用五声音阶作为总体调性参考，同时让实际变调后的目标音优先落在**当前和弦音**上，而不是只使用一套不随和弦变化的固定五声音阶。

## 3. 变调分组与布局约束

每个音节保留四个音高分组，排列顺序固定为：

```text
较高音  →  次高音  →  原始音频  →  较低音
```

- 横屏：四组从左到右排列，原始音频固定在第三列。
- 竖屏：四组从上到下排列，原始音频固定在第三行。
- `da / gou / jiao` 各自的第三组必须使用 `playbackRate = 1`，不得做任何变调。
- 第一、第二组的变调后实际基频必须高于对应原始音频的参考基频。
- 第四组的变调后实际基频必须低于对应原始音频的参考基频。

由于三段原始音频的基频并不一致，不能给它们共用同一组固定 `playbackRate`。每个音节都必须根据自己的实测基频单独计算倍率。

## 4. 当前原始音频的初步检测结果

初步分析使用逐帧 YIN 基频检测，并用频谱谐波峰和自相关峰进行交叉检查，以降低把二次谐波误认为基频、造成高八度误判的风险。

| 音频 | 初步主音高 | 逐帧特征 | 注意事项 |
| --- | --- | --- | --- |
| `da.wav` | B4 附近 | 约从 504 Hz 下降到 444 Hz | 音高轮廓明显下降，不能把整段视为绝对恒定音高 |
| `gou.wav` | 约 360 Hz | 主要位于 F4 与 F♯4 之间 | 原始音没有严格落在十二平均律音名中心 |
| `jiao.wav` | B4 附近 | 主体约 483–504 Hz | 主体比 `da` 稳定，但仍存在自然音高摆动 |

这些数据只记录初步结论。正式实现应以 `tools/` 中分析程序重新生成的报告为准，不应人工听辨后手填或直接采用整数音名。

语音样本本身带有音高曲线，当前 `playbackRate` 算法只能整体缩放这条曲线，不能把它拉平成恒定音符。因此“达到目标音”具体指：将高能量、高置信度的有声帧统计所得参考基频对准目标音；原有的升降调轮廓会被保留。

## 5. 十二平均律与变调倍率

以 A4 = 440 Hz 为基准，频率与 MIDI 音高的换算为：

```text
midi = 69 + 12 × log2(frequency / 440)

frequency = 440 × 2^((midi - 69) / 12)
```

将原始参考音高变到目标音高时：

```text
playbackRate
  = targetFrequency / sourceFrequency
  = 2^((targetMidi - sourceMidi) / 12)
```

例如，程序实测某个样本参考基频为 `sourceFrequency`，目标和弦音为 `targetFrequency`，则必须使用二者的实际比值；不能因为目标相差“大约几个半音”而直接手写 `1.25`、`1.5` 等近似倍率。

当前 Web Audio 播放链继续使用 `AudioBufferSourceNode.playbackRate`，不引入保持时长不变的独立移调算法。其结果是：

```text
变调后基频 ≈ 原始基频 × playbackRate
变调后时长 = 原始时长 ÷ playbackRate
```

所以升调会同时变快、变短，降调会同时变慢、变长。这属于当前算法的既定听感。

## 6. 和弦跟随的目标音选择原则

播放时间已经被量化到八分音符。目标音应按照**量化后的实际播放时刻**计算当前小节，而不是按照用户按下指针的瞬间计算，避免点击发生在和弦切换边界时选错和弦。

对每个音节及当前和弦：

1. 从当前三和弦的所有八度等价音中生成候选目标频率。
2. 第三组固定返回 `playbackRate = 1`。
3. 第二组选择高于原始参考基频的最近和弦音。
4. 第一组选择高于第二组的下一个和弦音。
5. 第四组选择低于原始参考基频的最近和弦音。
6. 使用实测基频计算每个音节各自的 `playbackRate`。
7. 对实际变调结果重新检测，确认第一组 > 第二组 > 原始组 > 第四组，并确认三组变调音落在预定和弦音允许误差内。

这种方法会让不同音节在同一个空间分组中使用不同的倍率，但它们最终会落到当前和弦的有效音上，符合“原始音高不同，按目标音反推倍率”的要求。

### 原始音频组的不可消除例外

第三组必须完全不变调，因此它的实际音高不可能同时成为 C、G、Am、F 四个和弦的和弦音。原始组在部分小节可能形成七音、挂留音、变化音或轻微失谐，这是“保留原始音频”与“所有点击都严格落在当前和弦音”无法同时满足时的明确例外。

## 7. 分析与验证工具约定

所有仅用于开发和验证的文件统一放在：

```text
tools/
```

包括：

- 音高检测程序；
- 变调倍率计算程序；
- 变调后音高复测程序；
- 分析报告；
- 调试用频谱、CSV、JSON；
- 临时生成的变调 WAV。

临时产物统一放在：

```text
tools/tmp/
```

网页运行代码和正式资源不得依赖 `tools/`。发布打包时可以直接排除整个 `tools/` 目录。

正式验证报告至少应记录：

- 原始音频逐帧 F0、RMS 和检测置信度；
- 被选作倍率计算依据的参考基频、MIDI 值和相对最近十二平均律音符的 cents 偏差；
- 每个和弦、音节、音高分组的目标音和计算倍率；
- 使用计算倍率生成的变调结果；
- 对变调结果重新检测得到的实际音高；
- 实际音高与目标音之间的 cents 误差；
- `da` 等非恒定音高样本的变调后音高范围，而不只记录单一平均值。

## 8. 资料来源

1. Open Music Theory, “Collections”：五声音阶可由大调的第 1、2、3、5、6 级构成；大调五声音阶移除了会形成半音的第 4、7 级。  
   <https://viva.pressbooks.pub/openmusictheory/chapter/collections/>

2. Open Music Theory, “Triads”：三和弦由根音、三音和五音构成，并说明大调与小调三和弦的音程结构。  
   <https://viva.pressbooks.pub/openmusictheory/chapter/triads/>

3. Open Music Theory, “Chord-Scale Theory”：和弦—音阶方法需要根据调性和和弦功能选择音阶，并建议在强拍使用和弦音连接和声进行。  
   <https://viva.pressbooks.pub/openmusictheory/chapter/chord-scale-theory/>

4. Alain de Cheveigné and Hideki Kawahara, “YIN, a fundamental frequency estimator for speech and music,” *Journal of the Acoustical Society of America*, 2002。  
   <https://pubmed.ncbi.nlm.nih.gov/12002874/>

5. Web Audio API 1.1 specification, `AudioBufferSourceNode`：`playbackRate` 与 `detune` 共同决定实际播放倍率，浏览器通过重采样播放缓冲区。  
   <https://webaudio.github.io/web-audio-api/#AudioBufferSourceNode>

6. MDN, `AudioBufferSourceNode.playbackRate`：`1.0` 为原速，小于 `1.0` 变慢，大于 `1.0` 变快，非 `1.0` 时会重采样音频。  
   <https://developer.mozilla.org/en-US/docs/Web/API/AudioBufferSourceNode/playbackRate>
